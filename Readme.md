# Credit Card Fraud Detection вЂ” SQL Analytics Engine

An end-to-end fraud detection project covering data engineering, advanced SQL
analytics, and rule-based fraud alerting вЂ” built on 6.3 million real-world
transaction patterns.

## Project Highlights

- Designed a normalized 4-table SQL Server database from a flat CSV
- Loaded 6.3M+ transactions using Python (pandas + SQLAlchemy)
- Wrote 8 advanced SQL queries uncovering real fraud patterns
- Built a 4-rule fraud alert engine using SQL `INSERT` statements
- Identified a **100% precision fraud detection rule** (zero false positives)

## Tech Stack

- **Database:** SQL Server 2022 (SSMS)
- **ETL:** Python (pandas, SQLAlchemy, pyodbc)
- **Analytics:** SQL (CTEs, window functions, UNION ALL, subqueries)
- **Coming next:** Machine Learning (scikit-learn), Dashboard

## Database Schema

4 normalized tables: `accounts`, `transactions`, `fraud_labels`, `fraud_alerts`.

| Table | Purpose |
|-------|---------|
| accounts | Unique sender/receiver account registry |
| transactions | Core transaction fact table (6.3M rows) |
| fraud_labels | Ground-truth fraud flags |
| fraud_alerts | Rule-based alerts generated by SQL |

## Key Findings

| Finding | Insight |
|---------|---------|
| **Fraud concentration** | 100% of fraud occurs in only 2 of 5 transaction types (TRANSFER, CASH_OUT) |
| **Balance drain rule** | Accounts fully drained to $0 are fraud **100% of the time** (8,008 cases, zero false positives) |
| **Hit-and-run pattern** | 8,185 fraud accounts used exactly once then abandoned вЂ” no repeat offenders |
| **Time pattern** | Fraud rate peaks at **22%** between 3вЂ“4 AM, vs <0.1% during business hours |
| **Financial impact** | $12B+ in fraudulent transactions identified across the dataset |

## Fraud Alert Engine

4 SQL-based detection rules populate the `fraud_alerts` table:

| Rule | Risk Score | Alerts Generated |
|------|-----------|-------------------|
| FULL_DRAIN | 0.95 | 8,008 (100% precision) |
| HIT_AND_RUN | 0.70 | 8,185 |
| NIGHT_TRANSFER | 0.85 | 4,750 |
| HIGH_AMOUNT | 0.75 | 130,507 |

4. Run `notebooks/load_data.py` to populate the database
5. Run the SQL scripts in `sql/` in numbered order

## Machine Learning вЂ” Fraud Detection Model

### Pipeline
SQL Server в†’ Python (feature engineering) в†’ train/test split в†’ Logistic Regression

### Feature Engineering
Built directly from SQL findings:

| Feature | Derived From |
|---------|-------------|
| `hour_of_day`, `is_night` | Query 6 вЂ” 3-4 AM fraud spike |
| `is_full_drain` | Query 4 вЂ” 100% precision rule |
| `balance_diff_orig`, `balance_diff_dest` | Balance change patterns |
| `error_balance_orig`, `error_balance_dest` | Mismatch between expected/actual balance |
| `type_TRANSFER`, `type_CASH_OUT`, etc. | One-hot encoded transaction type |

### Handling Class Imbalance
Fraud is only 0.129% of transactions. Used `class_weight='balanced'` in
Logistic Regression rather than naive accuracy optimization, which would
otherwise default to predicting "not fraud" every time.

### Baseline Model Results вЂ” Logistic Regression

| Metric | Score |
|--------|-------|
| Accuracy | 99.88% |
| Precision | 51.34% |
| **Recall** | **98.97%** |
| F1 Score | 0.676 |

**Confusion Matrix (test set):**
| | Predicted Legit | Predicted Fraud |
|---|---|---|
| **Actual Legit** | 1,269,340 | 1,541 |
| **Actual Fraud** | 17 | 1,626 |

> Only 17 fraud cases out of 1,643 were missed вЂ” a 98.97% catch rate,
> prioritizing recall over precision since missed fraud costs far more
> than a manual review false alarm.
