# Credit Card Fraud Detection вҖ” SQL Analytics Engine

An end-to-end fraud detection project covering data engineering, advanced SQL
analytics, and rule-based fraud alerting вҖ” built on 6.3 million real-world
transaction patterns.

## Project Highlights

- Designed a normalized 4-table SQL Server database from a flat CSV
- Loaded 6.3M+ transactions using Python (pandas + SQLAlchemy)
- Wrote 8 advanced SQL queries uncovering real fraud patterns
- Built a 4-rule fraud alert engine using SQL `INSERT` statements
- Identified a **100% precision fraud detection rule** (zero false positives)

## Tech Stack

- **Database:** SQL Server 2022 (SSMS)
- **ETL:** Python (pandas, SQLAlchemy, pyodbc)
- **Analytics:** SQL (CTEs, window functions, UNION ALL, subqueries)
- **Coming next:** Machine Learning (scikit-learn), Dashboard

## Database Schema

4 normalized tables: `accounts`, `transactions`, `fraud_labels`, `fraud_alerts`.

| Table | Purpose |
|-------|---------|
| accounts | Unique sender/receiver account registry |
| transactions | Core transaction fact table (6.3M rows) |
| fraud_labels | Ground-truth fraud flags |
| fraud_alerts | Rule-based alerts generated by SQL |

## Key Findings

| Finding | Insight |
|---------|---------|
| **Fraud concentration** | 100% of fraud occurs in only 2 of 5 transaction types (TRANSFER, CASH_OUT) |
| **Balance drain rule** | Accounts fully drained to $0 are fraud **100% of the time** (8,008 cases, zero false positives) |
| **Hit-and-run pattern** | 8,185 fraud accounts used exactly once then abandoned вҖ” no repeat offenders |
| **Time pattern** | Fraud rate peaks at **22%** between 3вҖ“4 AM, vs <0.1% during business hours |
| **Financial impact** | $12B+ in fraudulent transactions identified across the dataset |

## Fraud Alert Engine

| Rule | Risk Score | Alerts Generated |

| HIT_AND_RUN | 0.70 | 8,185 |

## Machine Learning вҖ” Fraud Detection Model

### Pipeline
SQL Server вҶ’ Python (feature engineering) вҶ’ 3 models trained and compared

### Feature Engineering
Built directly from SQL analytics findings:

| Feature | Source |
|---------|--------|
| `hour_of_day`, `is_night` | Query 6 вҖ” 3-4 AM fraud spike (22% rate) |
| `is_full_drain` | Query 4 вҖ” 100% precision SQL rule |
| `balance_diff_orig/dest` | Balance change pattern detection |
| `error_balance_orig/dest` | Mismatch between expected and actual balance |
| `type_*` | One-hot encoded transaction type (5 categories) |

### Handling Class Imbalance
Fraud = 0.129% of transactions. Used `class_weight='balanced'`
across all models вҖ” prevents the model defaulting to "predict legit
every time" which would give 99.87% accuracy but catch zero fraud.

### Model Comparison Results

| Metric | Logistic Regression | RF Full | RF Realistic |
|--------|--------------------|---------| -------------|
| Accuracy | 99.87% | 100.00% | 100.00% |
| Precision | 49.35% | 100.00% | **99.94%** |
| Recall | 99.21% | 99.45% | **99.45%** |
| F1 Score | 0.659 | 0.997 | **0.997** |
| False alarms | ~1,541 | ~15 | ~1 |
| Missed fraud | ~17 | ~2 | ~2 |

### Key ML Findings

1. **Logistic Regression** вҖ” strong recall (99.21%) but poor precision
   (49.35%), meaning too many false alarms for practical deployment.

2. **Random Forest Full** вҖ” near-perfect results, but feature importance
   analysis revealed `is_full_drain` was responsible for 37.7% of all
   decisions вҖ” a single SQL-derived rule dominating the model.

3. **Random Forest Realistic** вҖ” removed `is_full_drain` to test true
   model robustness. Results barely changed (99.94% precision, 99.45%
   recall), proving the model had genuinely learned fraud patterns
   across multiple features independently.

4. **Feature importance insight** вҖ” after removing `is_full_drain`,
   importance redistributed naturally to `error_balance_orig` (27.5%)
   and `balance_diff_orig` (24.3%), confirming the model captures the
   underlying account-draining pattern without relying on any single
   feature.

### Recommended Model
**Random Forest Realistic** вҖ” robust, generalizable, not dependent on
any single dominant feature. Catches 99.45% of fraud with a 99.94%
precision rate.
